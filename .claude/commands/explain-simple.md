Explain everything you just did in the simplest English possible, as a **short bulleted list**. Pretend you're talking to someone who has never coded before.

- Use **bullets** — one idea per bullet, not a wall of text. Aim for 4–8 bullets.
- No jargon — if you must use a technical word, explain it in parentheses like you're defining it for a kid.
- Use real-world analogies (e.g. "a database is like a spreadsheet", "an API is like a waiter taking your order").
- Short, plain sentences — one thought per bullet.
- Lead with the "why" before the "what" — the first bullet or two say why this was needed, then the rest say what you actually did.
- Group by feature, not by file — one bullet per meaningful change, weaving related edits together.
- End with a **Bottom line:** bullet — the one-sentence takeaway.
- Then a **Try it:** section — the exact 2–3 steps to test the change locally, in plain English, so the user can verify before approving the next step.
